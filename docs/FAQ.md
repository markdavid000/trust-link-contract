# Frequently Asked Questions

Answers to common questions contributors encounter when working on TrustLink.

---

## Local Testing

### How do I run the full test suite?

```bash
cargo test --manifest-path contracts/escrow/Cargo.toml
```

Run from the repository root. All unit and integration tests for the escrow contract are executed in one pass.

### How do I run a single test?

```bash
cargo test --manifest-path contracts/escrow/Cargo.toml <test_name>
```

Replace `<test_name>` with the exact test function name, e.g. `test_mark_shipped_transitions_state`.

### How do I update or regenerate test snapshots?

Delete the stale snapshot file under `contracts/escrow/test_snapshots/` and re-run the tests. The Soroban test harness recreates any missing snapshot automatically on the next successful run.

## Common Errors

### `error[E0428]: the name â€¦ is defined multiple times`

This usually means the `lib.rs` contains a duplicate type definition â€” most often `EscrowState`. Remove the older definition (typically the one without the `Shipped` variant) and keep only the canonical version.

### `error[E0433]: failed to resolve: could not find 'Escrow' in the crate root`

The `#[contract]` macro on `pub struct Escrow;` is missing from `lib.rs`. Confirm the struct declaration and its attribute are present.

### `cannot find function 'load_escrow' / 'save_escrow'`

The private helper functions (`load_escrow`, `save_escrow`, `load_dispute`, `save_dispute`, `deduct_and_transfer`) are missing from `lib.rs`. These are defined in the same file above the `#[contractimpl]` block. A bad merge can silently drop them.

### `error[E0432]: unresolved import crate::EscrowClient`

The `EscrowClient` type is auto-generated by the `#[contractimpl]` macro. If it is missing the contract struct itself is missing or the macro failed to expand. Check that `#[contract]` and `#[contractimpl]` are both present and that there are no duplicate type definitions causing compile errors earlier in the file.

### `already initialized` panic in tests

Each test environment is isolated, but if you accidentally call `initialize` twice on the same contract instance in a single test the contract panics. Use a fresh `Env` per test or the `setup_contract` helper from `test_helpers.rs` which sets up a clean instance.

### `contract paused` panic in tests

The contract is paused via `pause_contract()` and a mutating function was called. Either call `unpause_contract()` first, or create a fresh `Env` that has not been paused.

### `not initialized` panic in tests

`initialize` was never called on the contract before using it. Always call it (or use `setup_contract`) at the start of each test.

## Sandbox Environment

### Which Rust toolchain is required?

The project targets `wasm32-unknown-unknown` for on-chain deployment and uses stable Rust. Install the target with:

```bash
rustup target add wasm32-unknown-unknown
```

Tests run against the native target so no extra toolchain setup is needed just to run `cargo test`.

### How do I check for warnings without running tests?

```bash
cargo check --manifest-path contracts/escrow/Cargo.toml
```

### Why does the contract use `#![no_std]`?

The Soroban runtime does not provide a standard library in the WASM environment. All code must be `no_std` compatible. Test helpers can still use `std` because they are gated behind `#[cfg(test)]`.

## Contributing

### Where do I find open issues?

Browse the [GitHub Issues](../../issues) tab. Issues labelled `good first issue` are suitable starting points.

### Do I need to open a PR to add tests only?

Yes. Every code change â€” including new tests â€” must go through a pull request so CI can verify compilation and test passage.

### What should a PR description include?

- A short summary of what changed and why.
- A reference to the issue being closed using `closes #<number>`.
- A test plan describing what was verified manually or automatically.
